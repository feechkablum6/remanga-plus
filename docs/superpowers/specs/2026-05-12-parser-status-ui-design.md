# Статусы парсера вместо скелетон-загрузки

## Проблема

При поиске платной главы пользователь видит 5 бессмысленных shimmer-линий и статичный текст «Запускаем parser-server…». Непонятно, что происходит внутри — парсер уже подключился? Какой источник проверяется? Почему долгая загрузка?

## Решение

Заменить скелетон на статус-блок с SVG-иконками и чипами провайдеров. Иконки морфируются из одной стадии в другую плавно (SVG path interpolation ~500ms). Чипы провайдеров тоже содержат мини-SVG и морфируются при смене статуса.

Параллельный парсинг (другой агент переделывает парсер на async) — все 4 провайдера опрашиваются одновременно, статусы обновляются через polling.

## Структура UI

Центр экрана — статус-блок:

```
┌────────────────────────────────┐
│       [SVG иконка 52px]        │  ← морфится: ⚡ → 🔍 → ✓ / ⚠
│     Текст текущей стадии       │  ← «Подключение…» / «Ищем главу…» / «Глава найдена»
│                                │
│  [chip] [chip] [chip] [chip]   │  ← провайдеры: мини-SVG + текст, морфятся
└────────────────────────────────┘
```

Под статус-блоком — страницы главы (рендерятся заранее, скрыты за карточкой).

## Фазы

### 1. Подключение
- Иконка: ⚡ (молния), рамка серая `#334155`
- Текст: «Подключение…»
- Чипы: не показываются (парсер ещё не запущен)

### 2. Поиск
- Иконка morph: ⚡ → 🔍 (лупа), рамка синяя `#1d4ed8`, pulse-анимация
- Текст: «Ищем главу…»
- Чипы: 4 провайдера, каждый в статусе «поиск» — серая рамка `#334155`, мини-SVG 🔍
- По мере ответов провайдеров чипы морфятся:
  - Не найдено: рамка красная `#450a0a`, мини-SVG ✕, текст «Mangabuff»
  - Ошибка провайдера: рамка красная `#450a0a`, мини-SVG ✕, текст «Senkuro — ошибка»
  - Найдено: рамка зелёная `#16a34a`, мини-SVG ✓, текст «Senkuro»

### 3а. Успех
- Иконка morph: 🔍 → ✓, рамка зелёная `#16a34a`, фон `#052e16`
- Текст: «Глава найдена»
- Чип: нашедший провайдер зелёный, не успевшие — серые «—»
- Пауза 1.5с → fade-out всего статус-блока 400ms → обнажаются страницы главы
- Страницы главы уже отрендерены в DOM под статус-блоком (абсолютное позиционирование или z-index), скрыты за ним. При fade-out блок становится прозрачным и страницы проступают.

### 3б. Ошибка (все провайдеры не нашли)
- Иконка morph: 🔍 → ⚠, рамка красная `#dc2626`, фон `#1c0a0a`
- Текст: «Глава не найдена»
- Подтекст: «Ни один источник не содержит эту главу»
- Чипы: все красные с ✕
- Кнопки: «Повторить» + «Карточка Remanga»

## SVG-морфинг

Все иконки — SVG с одинаковым числом control points для корректной CSS/JS интерполяции. Набор иконок:

| Иконка | Стадия | Цвет рамки |
|--------|--------|------------|
| ⚡ Молния | Подключение | `#334155` |
| 🔍 Лупа | Поиск | `#1d4ed8` |
| ✓ Галочка | Успех | `#16a34a` |
| ✕ Крестик | Не найдено | `#64748b` |
| ⚠ Треугольник | Ошибка | `#dc2626` |

Чипы используют те же SVG, масштабированные до ~16px.

## Polling-протокол

Парсер (переделывается другим агентом) отдаёт промежуточные статусы через polling. Content script опрашивает parser-server и обновляет UI по мере поступления ответов от провайдеров.

Данные от парсера (ожидаемый формат polling-ответа):
```json
{
  "status": "in_progress" | "completed",
  "providers": {
    "mangabuff": { "status": "searching" | "not_found" | "error" | "found", "detail?": "..." },
    "senkuro": { "status": "searching" | "not_found" | "error" | "found", "detail?": "..." },
    "inkstory": { "status": "searching" | "not_found" | "error" | "found", "detail?": "..." },
    "teletype": { "status": "searching" | "not_found" | "error" | "found", "detail?": "..." }
  },
  "result?": { ... }
}
```

Content script: poll с интервалом ~500ms до `status: "completed"`.

## Feed-режим: подгрузка следующей главы

При подгрузке следующей главы внизу ленты — мини-версия того же статус-блока: маленькая SVG-иконка + текст стадии + мини-чипы провайдеров. Тот же SVG morph, визуальное единство с главным статусом.

## Убирается

- 5 скелетон-линий (`data-rre-control="premium-free-skeleton-line"`)
- Статичный текст «Запускаем parser-server, ищем подходящий источник и подготавливаем章节 к чтению.»
- CSS shimmer-анимация `rre-premium-skeleton`

## Зависимости

- Парсер должен поддерживать polling со статусами провайдеров (другой агент)
- Content script должен рендерить страницы главы под статус-блоком до fade-out
